# 排期优化：产品需求阶段与验收冲突解决新逻辑

## 一、当前模型的局限

### 1.1 当前字段定义

| 引擎字段 | 飞书字段 | 含义 |
|---------|---------|------|
| `review_date` | 需求评审 | 技术评审日期，+1wd = 研发可开始 |
| `design_start` | 需求开始 | PM需求设计期起点 |
| `design_end` | 需求结束 | PM需求设计期终点 |

### 1.2 当前冲突逻辑

```python
# 当前验收调度中的冲突处理
验收就绪(test_end + 1wd) → 寻找可用窗口:
  与PM设计期重叠:
    如果 gap(dev→test→accept) > max_gap(5wd):
      → 验收插队(设计期被侵占,记录preempted_days)
    否则:
      → 验收等待,后推到设计期结束
  与PM其他验收重叠:
      → 验收等待,后推
```

### 1.3 当前模型的问题

1. **需求阶段不可调**：`design_start~design_end` 是固定输入，排期过程中不移
2. **验收只能等待或插队**：没有"验收优先、需求后移"的调度策略
3. **需求阶段不占用PM资源池**：只做冲突检测，不做资源调度
4. **后续任务不自动顺延**：前序验收延时不传播到后续任务的需求阶段

## 二、新字段模型

### 2.1 新增/变更字段

| 字段 | 类型 | 含义 | 引擎变量 |
|------|------|------|---------|
| **澄清开始** | date | 产品澄清阶段开始 | `clarify_start` |
| **澄清结束** | date | 产品澄清阶段结束 | `clarify_end` |
| **评审开始** | date | 方案评审阶段开始 | `tech_review_start` |
| **评审结束** | date | 方案评审阶段结束 | `tech_review_end` |
| **需求开始** | date | (保留)需求设计阶段开始 | `design_start` |
| **需求结束** | date | (保留)需求设计阶段结束 | `design_end` |
| **技术评审** | date | (原需求评审)技术评审日期 | `tech_review` = 旧 `review_date` |

### 2.2 产品需求阶段定义

```
产品需求阶段 = [clarify_start, clarify_end] ∪ [tech_review_start, tech_review_end]
             ∪ [design_start, design_end]
技术评审 = tech_review (原需求评审) → 需求阶段的最终时间点
研发开始 = next_working_day(tech_review)
```

### 2.3 关键规则

- **产品需求阶段消耗PM资源**：PM在同一时间只能做一个任务的需求阶段或验收
- **技术评审是需求阶段的最后节点**：tech_review_date = 需求阶段的最晚日期
- **需求阶段的工期** = `working_days_between(需求阶段起点, 需求阶段终点)`
- **需求阶段是排期结果而非固定输入**：Base中填写的澄清/评审/需求起止是"计划值"，
  引擎可动态调整（类似研发/测试/验收的排期）

## 三、新冲突解决逻辑

### 3.1 核心原则

```
验收优先级 > 需求阶段优先级
需求阶段可被验收"打断"，延后执行
延后的需求阶段会传播到该PM的后续任务
```

### 3.2 调度算法

```
对每个PM p:
  维护两个队列:
    req_queue[p] = [(task1的需���阶段), (task2的需求阶段), ...] 按技术评审日期排序
    accept_ready[p] = [(taskA的验收), (taskB的验收), ...] 按就绪时间排序

逐日模拟中，对每个PM:
  如果 accept_ready[p] 非空 AND PM空闲:
    → 从 accept_ready[p] 取就绪任务,安排验收
    → PM忙碌 = 验收时间段
    验收结束后 → PM重新可用
    
  如果 accept_ready[p] 非空 AND PM在进行需求阶段:
    → 判断当前正在做的需求阶段是否属于该验收任务自身
      ├─ 是 → 验收等该任务需求阶段结束后开始 (需求阶段优先完成)
      └─ 否 → 验收可打断 → 中断当前需求阶段,开始验收
              (被打断的需求阶段搁置,其任务的技术评审→研发→测试全部后移)
```

### 3.3 具体场景逻辑

#### 场景1.1：验收就绪时，PM未进入该任务的需求阶段

```
条件: 任务A的test_end + 1wd ≤ sim_day
      && taskA的需求阶段尚未开始
      
操作:
  1. 直接开始验收A（占PM时间）
  2. 验收A结束后 + 1wd:
     a. 开始需求阶段A（占PM时间）
     b. 需求阶段结束 = 技术评审A
     c. 研发A开始 = 技术评审A + 1wd
  3. 该PM的后续任务B/C的需求阶段全部顺延
```

**顺延传播链**：
```
验收A [占用PM] → 需求A后移 → 技术评审A后移 
→ 研发A后移 → 测试A后移 
→ 需求B后移 → 技术评审B后移 → 研发B后移 ...
```

#### 场景1.2：验收就绪时，PM已在做该任务的需求阶段

```
条件: 任务A的test_end + 1wd ≤ sim_day
      && taskA的需求阶段正在进行中

操作:
  1. 验收A的开始时间 = 需求阶段A的结束时间 + 1wd
     (即等待当前需求阶段做完)
  2. 验收A结束后PM空闲
  3. 后续任务B/C的需求阶段顺延
```

#### 场景1.3（扩展）：验收就绪时，PM在做另一个任务的需求阶段

```
条件: 任务A的test_end + 1wd ≤ sim_day
      && PM在做任务B的需求阶段(B ≠ A)

操作:
  1. 验收A打断需求阶段B
  2. 记录需求阶段B已消耗的天数
  3. 验收A开始
  4. 验收A结束后 +1wd → 恢复需求阶段B（从断点继续）
  5. 任务B的后续研发/测试/验收全部顺延
```
