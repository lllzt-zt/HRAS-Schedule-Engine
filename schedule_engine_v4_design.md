# HRAS 排期引擎 v4 完整逻辑说明

> 本文档将 v3 引擎的原有逻辑与本次新增的"产品需求阶段全自动排期"逻辑合并，
> 作为更新 SKILL.md 和 schedule_logic.md 的依据。

---

## 一、整体任务管线（不变）

每个Task的完整生命周期（**同Task内顺序固定，不可调换**）：

```
[需求阶段] → [技术评审] → [研发] → [测试] → [验收]
   ↑PM时间       ↑最后      ↑开发池   ↑测试池    ↑PM时间
                 时间点      非PM      非PM
```

**不变的公式**：
- `技术评审 + 1wd = 研发可开始`
- `研发人天 = 从产研比计算（1:2→迭代×10，1:3→迭代×15，无→0跳过）`
- `测试人天 = 读Base公式字段，空则降级迭代×6`
- `验收人天 = 读Base公式字段，空则降级迭代×2.5`
- `研发/测试/验收工作日 = ceil(人天 / 人数)`
- 研发池（1-6 + K1/K2）、测试池（1-4）分配逻辑不变

## 三、输入规则（新增）

### 3.1 首Task识别（每个PM独立）

**每个PM各自拥有独立的首Task**，识别规则：

1. 扫描该PM名下所有任务
2. 找出有技术评审日期（非空）的任务
3. 其中技术评审日期最早且非历史（≥today）的为该PM的首Task
4. 该PM下其余任务（包括那些Base中有技术评审日期但比首Task晚的）均视为后续Task

### 3.2 首Task的输入

PM只给定首Task的：
- 完整需求周期（clarify_start/end, tech_review_start/end, design_start/end）
- 技术评审日期（tech_review_A）—— 这是影响后续排期的唯一锚点

**需求阶段开始时间**：
```
如果 tech_review_A 倒推 req_wd_A 后 ≥ today:
  req_start_A = tech_review_A - req_wd_A（倒推req_wd_A个工作日）
否则:
  req_start_A = today
  tech_review_A = today + req_wd_A（顺延）
```

### 3.2 后续Task（Task B/C/...）

PM不给定任何日期，全部由引擎生成：
- 需求阶段开始时间 = 由PM Scheduler决定（详见第5节）
- 技术评审日期 = 需求阶段结束日（联动后移）
- 研发/测试/验收开始时间 = 按引擎原有公式计算

## 四、PM资源模型

### 4.1 PM时间线的两条任务类型

| 类型 | 含义 | 占用PM |
|:----:|:----:|:------:|
| 需求阶段 | 澄清+评审+需求设计 | ✅ 独占PM，串行 |
| 验收 | 产品验收 | ✅ 独占PM，串行 |

研发/测试走研发/测试资源池，**不占用PM时间**，可以与PM任务**并行执行**。

### 4.2 PM任务优先级规则

```
所有PM待办（需求阶段 + 验收）按各自的"开始时间"排序，
开始时间最早的先做，
如果开始时间相同 → 需求阶段优先。
```

**"开始时间"的定义**：
- **需求阶段**的开始时间 = 该Task的计划需求开始时间（由引擎计算得出）
- **验收**的开始时间 = `test_end + 1wd`（验收就绪的时间）

## 五、PM Scheduler（新增，引擎核心改动）

### 5.1 总体流程

```
PM Scheduler 是 schedule_all() 主循环的一部分。

在每个模拟日，对于每个PM：
  
  ① 检入：检查该PM所有Task，标记新就绪的验收(test_end + 1wd ≤ sim_day)
  
  ② 如果PM空闲（无正在进行的需求阶段或验收）:
     a. 收集所有待办事项
        - 已就绪的验收（test_end + 1wd ≤ sim_day）
        - 未开始的需求阶段
     b. 按"开始时间"排序（最早的先做，同时优先需求）
     c. 取第一个 → PM开始执行
     
  ③ 如果PM忙碌:
     a. 不打断当前任务
     b. 等当前任务完成后（模拟日推进到结束日），回到 ②
```

### 5.2 验收就绪时的两种场景（用Task A/B/C表述）

条件：Task A 验收就绪（test_A_end + 1wd ≤ sim_day），检查PM状态：

> 注意：同Task内需求阶段在前、验收在后，所以"PM在做Task A自身需求阶段时验收A就绪"在结构上不可能发生，不列入。

```
                  ┌─ 1.1 PM空闲
验收A就绪 ──┤
                  │
                  └─ 1.3 PM在做其他Task B的需求阶段
                        → 不打断Task B
                        → 等Task B需求阶段结束（B的技术评审）+ 1wd
                        → 验收A开始
```

#### 场景 1.1：PM空闲 🟢

| 条件 | 验收A就绪时，PM没有任何正在做的需求阶段或验收 |
|:----:|:-------------------------------------------:|
| **操作** | ① 验收A立刻开始<br>② 验收A结束 → PM空闲 → 回到PM Scheduler决策循环 |
| **后续影响** | 如果验收A结束时间 ≥ Task B的计划需求开始时间 → B的需求阶段顺延（连锁传播） |

#### 场景 1.3：PM在做Task B的需求阶段 🔴

| 条件 | 验收A就绪时，PM正在做Task B的需求阶段（B ≠ A） |
|:----:|:--------------------------------------------:|
| **操作** | ① **不打断** Task B的需求阶段<br>② 等Task B需求阶段结束（B的技术评审）<br>③ B的技术评审 + 1wd → 验收A开始 |
| **后续影响** | 验收A结束后PM空闲 → 回到PM Scheduler决策循环 |

### 5.3 后续Task的链式依赖

```
Task B的需求阶段开始时间：
  = A的技术评审 + 1wd（最早PM空闲时间）
  （PM在A的技术评审后立即空闲，因为A的研发/测试不占用PM）

Task C的需求阶段开始时间：
  = PM Scheduler在验收A结束后，按优先级规则决定的下一件事的开始时间
  通常情况下：验收A结束后如果有空闲，且验收B还没就绪，则开始reqC
```

## 六、传播效应

当需求阶段后移时，在同一PM上的连锁反应：

```
验收A占用PM时间 → Task B需求阶段后移
→ Task B技术评审后移（联动）
→ Task B研发开始后移（技术评审+1wd，公式不变）
→ Task B测试后移 → Task B验收后移
→ Task C需求阶段后移 → ... 以此类推
```
