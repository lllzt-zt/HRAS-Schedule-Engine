# HRAS 项目排期引擎 v4

## ⚠️ 数据铁律（最高优先级）

**每次排期必须重新拉取飞书在线表最新数据。严禁从历史会话目录直接读取缓存文件。**

- ❌ 禁止：读取历史会话缓存
- ❌ 禁止：复用上次运行留下的本地 JSON 文件
- ✅ 必须：通过 `lark-cli base +record-list` 实时拉取飞书在线表数据
- ✅ 必须：先拉数据，再跑引擎，最后写回 — 三步串行，不可跳过第一步

## 多期排期概览

引擎支持**一/二/三期**排期，各期关系：

```
一期 → 排期固定，22个任务，日期不可改动
  ↓  各PM的一期结束时间不同，二期从一期完成后开始
二期 → 锚点任务（已固定日期）保留，新任务由引擎生成
  ↓
三期 → 所有任务由引擎生成，按分期优先级排在二期之后
```

**关键原则**：一期排期为锚，二期/三期在一期基础上并行推进。每个PM独立考虑其一期结束时间。

## 用途

为 HRAS 敏捷项目提供端到端排期能力：从飞书多维表格读取实时任务数据，应用时间驱动跨模块并行调度算法与 PM Scheduler，生成排期报告并写回飞书。

## v4 核心能力

- **时间驱动模拟**：逐工作日推进，同时为所有就绪任务分配资源
- **跨模块并行**：优先级仅仲裁同槽位竞争，空闲资源不囤积
- **PM Scheduler**：需求阶段和验收共享 PM 时间线，按开始时间排序，先到先做
- **首Task锚定**：**每位PM独立识别首Task**，仅首Task有技术评审日期，后续Task全部由引擎生成
- **需求阶段全自动排期**：需求阶段从固定输入变为可调度资源，后移时技术评审联动
- **验收与需求阶段冲突管理**：验收不打断需求阶段，按 FIFO 排序决策
- **A/B绩效资源**：绩效研发使用独立资源 A/B（原101/102），空闲时自动共享给普通任务
- **动态槽位分配**：不依赖 Base 预填值，每次按池实时空闲自动分配
- **历史已完成排除**：飞书表中"已完成=是"的任务自动排除
- **澄清日期自动推导**：有需求开始但无澄清日期的任务，引擎自动从需求开始倒推算澄清时间
- **末尾任务加人**：瓶颈PM的末尾任务可增加研发/测试人数，直接提升最终完成日
- **多期并行排期**：一期固定不可动，二期/三期按分期优先级+模块优先级 + 每PM独立起始日排期
- **分期优先排序**：排序规则为 `分期(二期→三期) > 模块优先级 > 记录顺序`
- **产测比计算**：标准测试人天 = 迭代数 × 产测比第二数值 × 5，产测比为必填字段

## 触发场景

- 用户报告资源变化（如"研发少了一个人"）
- 用户关键词：排期、schedule、优化排期、重新排期、资源变动

## 一、整体任务管线

每个 Task 的完整生命周期（同Task内顺序固定，不可调换）：

```
[需求阶段] → [技术评审] → [研发] → [测试] → [验收]
   ↑PM时间      ↑最后      ↑开发池  ↑测试池   ↑PM时间
                 时间点     非PM     非PM
```

- 需求阶段（澄清+需求设计+业务评审）= **占用 PM 时间**
- 技术评审 = 需求阶段的最后时间点（业务评审结束+1wd），是唯一影响后续排期的锚点
- 研发/测试 = 走开发池/测试池，**不占用 PM 时间**
- 验收 = **占用 PM 时间**

## 三、输入规则

### 3.1 首Task识别（每个PM独立）

**每个PM各自拥有独立的首Task**，识别规则：

1. 扫描该PM名下所有任务
2. 找出有技术评审日期（非空）或有完整需求阶段日期（澄清结束不为空）的任务
3. 选择其中开始时间最早的为该PM的首Task
4. 首Task的技术评审日期 < today → 排除（历史任务）
5. 该PM下其余任务均为后续Task

### 3.2 首Task无技术评审时的推导

当首Task没有技术评审日期但有需求结束日期时：
```
业务评审开始 = 需求结束 + 1wd
业务评审结束 = 业务评审开始 + 业务评审人天(工作日)
技术评审 = 业务评审结束 + 1wd
```

### 3.3 首Task无澄清日期时的推导

当首Task有需求开始但没有澄清日期时：
```
澄清总工期 = ceil(迭代数 × 3)
澄清结束 = 需求开始的前一个工作日
澄清开始 = 从澄清结束倒推(澄清总工期-1)个工作日
```

### 3.4 后续Task（全自动生成）

PM不给定任何日期，全部由引擎的PM Scheduler生成：
- 需求阶段开始时间 = 由 PM Scheduler 根据PM空闲时间决定
- 技术评审日期 = 业务评审结束 + 1wd
- 研发/测试/验收开始时间 = 按引擎公式计算

### 3.5 后续Task排序规则

后续Task按以下顺序处理：
1. 分期优先级（二期=0 > 三期=1）
2. 模块优先级（绩效>人事>考勤>薪酬>HRONE基础建设>流程平台）
3. 同模块同分期按Base表中的原始记录顺序

## 四、过滤规则

| 条件 | 处理 | 说明 |
|------|:----:|:----:|
| 已完成 = 是 | 排除 | 历史已完成任务 |
| 分期 = 持续迭代 | 排除 | 持续迭代不参与排期 |
| 首Task技术评审 < today | 排除 | 历史锚点任务 |
| 技术评审为 null 的后续Task | 正常 | 由引擎排期生成 |

## 五、资源池

### 5.1 资源池定义

| 池 | 资源 | 容量 | 说明 |
|---|------|:----:|------|
| 普通研发 | 1,2,3,4,5,6 | 6人 | 按研发人数自动分配空闲槽位 |
| 绩效研发 | **A, B** | 2人 | 绩效任务独占（标记"独立团队"） |
| 测试 | 1,2,3,4 | 4人 | 按测试人数自动分配空闲槽位 |
| PM时间线 | 无槽位 | 1人 | 需求阶段+验收串行，独占PM |

### 5.2 资源标识

- 绩效研发资源在报告和飞书表中用 **A/B** 标识（原101/102）
- 普通研发资源用 **1-6** 标识
- 当绩效A/B空闲且无绩效任务等待时，自动开放给普通任务使用（K1/K2共享机制）

### 5.3 动态分配规则

```
任务就绪 → 检查所需人数 → 从池中查找N个空闲槽位 → 锁定 → 写回
```

- **研发阶段**：非绩效任务从1-6 + 空闲A/B 动态分配
- **研发阶段**：绩效任务固定使用 A/B
- **测试阶段**：所有任务从1-4动态分配
- **槽位竞争**：优先级仅仲裁同槽位竞争，空闲资源不囤积

## 六、工时计算

### 6.1 研发人天

| 产研比 | 计算公式 |
|:------:|:--------:|
| `1:2` | 迭代数 × 10 |
| `1:3` | 迭代数 × 15 |
| 无 | 0（跳过研发阶段） |

### 6.2 测试人天（产测比公式）

从 v4 起，标准测试人天不再读取Base值，改用产测比计算：

```
标准测试人天 = 迭代数 × 产测比第二数值 × 5

示例:
  产测比 1:1   → 迭代数 × 1 × 5
  产测比 1:0.8 → 迭代数 × 0.8 × 5
  产测比 1:1.1 → 迭代数 × 1.1 × 5
```

> ⚠️ 产测比为**必填字段**，为空时引擎报错终止排期。

### 6.3 验收人天

直接读 `standard_accept_md`（公式字段），空时降级 迭代数 × 2.5

### 6.4 工作日转换（统一使用 math.ceil）

| 阶段 | 公式 |
|:----:|:----:|
| 研发工作日 | ceil(研发人天 / 研发人数) |
| 测试工作日 | ceil(测试人天 / 测试人数) |
| 验收工作日 | ceil(验收人天 / 1) |

## 七、PM Scheduler（v4核心）

### 7.1 主循环逻辑

```
PM Scheduler 在每个模拟日执行：

① 检入已就绪的验收(test_end + 1wd ≤ sim_day)
② 检查该PM的待办事项：
   a. 已就绪的验收
   b. 未开始的需求阶段
③ 按"开始时间"排序，取第一个执行
   开始时间相同 → 需求阶段优先
```

### 7.2 两种场景

**场景1：PM空闲时验收就绪 🟢**
```
验收A就绪，PM空闲 → 验收A立刻开始
验收A结束 → PM空闲 → 下一待办
```

**场景2：PM在做其他任务的需求阶段 🔴**
```
验收A就绪，PM正在做TaskB的需求阶段
→ 不打断TaskB
→ 等TaskB需求阶段结束（B技术评审）+ 1wd
→ 验收A开始
```

### 7.3 链式依赖

```
Task B的需求阶段开始 = A的技术评审 + 1wd（最早PM空闲时间）
Task C的需求阶段开始 = PM Scheduler 在验收A结束后决定
```

### 7.4 后续Task需求阶段生成

```
澄清总工期 = ceil(迭代数 × 3)
  前1/3非PM时间，后2/3PM时间
PM可开始时间 → 倒推算澄清开始 → 推算澄清结束
   ⚠️ 倒推算的澄清开始不得侵入上一任务PM占用期
需求开始 = 澄清结束 + 1wd
需求结束 = 需求开始 + 需求人天
业务评审开始 = 需求结束 + 1wd
业务评审结束 = 业务评审开始 + 业务评审人天
技术评审 = 业务评审结束 + 1wd
```

### 7.5 PM时间重叠防护机制

引擎通过 `pm_req_busy_until` 字典持久追踪每个PM最后需求阶段的结束日，防止后续任务的澄清开始日期倒推时侵入前一任务的PM占用期：

```
前一任务的PM需求阶段结束 → pm_req_busy_until[PM] = 结束日
下一任务澄清开始倒推 → 若 ≤ pm_req_busy_until → 顺延到结束后
下一任务PM可开始时间 → 自动后移至安全起始日
```

## 九、多期排期约束

### 9.1 一期排期（固定锚点）

一期22个任务排期**已固定**，任何排期操作不可修改一期日期：
- 一期最后完成日：**11/02**（刘观福·调薪验收结束）
- 一期PM最后结束时间各不相同，各PM独立计算

### 9.2 二期/三期排期原则

1. **首Task锚定**：二期第一个有原始日期的任务为首Task，日期固定不可动
2. **每PM独立起始**：各PM的二期从该PM一期最后占用结束日开始，不等待其他PM
3. **需求阶段不清除**：只清除有旧值的二期"新任务"的需求阶段日期，让引擎重新生成
4. **写回表不同**：一/二/三期均写入新表 `tbl5jDThuT51h4II`，非原始排期表

### 9.3 排序规则

```
排序键 = (
  首Task优先,               # 有固定日期的锚点排最前
  分期(二期=0, 三期=1),     # 全部二期排完再排三期
  模块优先级,                # 同分期内按模块排
  原始记录顺序               # 同模块同分期按表中顺序
)
```

## 十、基础约束条件

1. 研发开始 = 技术评审 + 1 工作日（不变）
2. 同一人/槽位不能同日处理两个任务
3. 周末和法定假日非工作日；调休补班为工作日
4. PM验收不能与其他PM活动（需求阶段或验收）重叠
5. 研发人天从产研比计算；测试/验收人天读表
6. **双软间隔**：dev→test + test→accept 均 max-gap=5wd，超时提升紧急度
7. **动态槽位**：引擎按池实时空闲情况从 1-6 + A/B 分配
8. **A/B共享**：绩效无研发任务时 A/B 自动纳入普通研发池
9. **测试池共享**：所有任务共用 1-4 测试池
10. **PM调度**：验收不打断需求阶段，按开始时间排序
11. **技术评审联动**：需求阶段后移 → 技术评审同步后移
12. **PM时间禁止重叠**：同一PM的需求阶段之间不可重叠，由 `pm_req_busy_until` 机制确保

## 十一、排期后冲突自动检查

每次排期写回后必须执行冲突检查，确保排期结果无冲突。

### 11.1 运行方式

```bash
python validate_schedule.py
```

### 11.2 检查项

| # | 检查项 | 规则 | 级别 |
|:-:|:------|:----|:----:|
| 1 | **PM时间重叠** | 同一PM的需求阶段、验收、需求vs验收之间不可有日期重叠 | error |
| 2 | **模块优先级顺序** | 同PM同分期内，任务按模块优先级升序排列（绩效<人事<考勤<薪酬<HRONE<流程） | error |
| 3 | **阶段顺序合理性** | 澄清start≤澄清end≤需求start≤需求end≤评审start≤评审end | error |
| | | 研发start≤研发end < 测试start≤测试end < 验收start≤验收end | error |

### 11.3 输出示例

```
▶ 检查1: PM时间重叠
  ✅ 刘观福: 无时间重叠
  ✅ 许大庆: 无时间重叠
  ✅ ... (所有PM)

▶ 检查2: 模块优先级顺序
  ✅ 所有PM模块顺序正确

▶ 检查3: 阶段顺序合理性
  ✅ 所有任务阶段顺序正确

✅ 全部通过！3/3 项检查通过，无冲突
```

### 11.4 处理方式

- 全部通过（exit code 0）：排期结果有效，可提交
- 发现冲突（exit code > 0）：**必须修复**，问题解决前不可作为最终排期
