### 1.2 过滤规则（v4调整）

| 规则 | 处理 | 说明 |
|------|------|:----:|
| 规划预留取消 = true | 排除 | 不变 |
| 技术评审为 null | **不过滤** | 后续Task正常状态，由引擎生成 |
| 技术评审 < 当天 且 为首Task | 排除（历史任务） | 仅限首锚点Task |

**首Task识别（v4按PM分组）**：
- 每个PM独立识别自己的首Task
- 扫描该PM下所有任务，有技术评审日期且最早（≥today）的为首Task
- 该PM下其余任务（即使Base中有技术评审值但比首Task晚）均为后续Task，由引擎重新计算

## 二、工时计算

### 2.1 研发人天（不变）

```
从产研比计算:
  1:2 → 迭代数 × 10
  1:3 → 迭代数 × 15
  无   → 0（跳过研发阶段）

对比飞书表中的标准研发人天({standard_dev_md}):
  如不一致 → 标记 dev_man_days_changed = true，使用计算值
```

### 2.2 测试人天（不变）

直接读 `standard_test_md`（公式字段），空时降级 迭代数 × 6

### 2.3 验收人天（不变）

直接读 `standard_accept_md`（公式字段），空时降级 迭代数 × 2.5

### 2.4 需求阶段工期（v4新增）

```
req_wd = working_days(clarify_start, clarify_end)
       + working_days(tech_review_start, tech_review_end)
       + working_days(design_start, design_end)
```

> req_wd 的具体换算公式（从迭代数计算）后续补充。

### 2.5 工作日转换

```
研发工作日 = ceil(研发人天 / 研发人数)
测试工作日 = ceil(测试人天 / 测试人数)
验收工作日 = ceil(验收人天 / 1)
需求工作日 = req_wd（固定工期）
```

## 三、资源池

### 3.1 研发资源池（不变）

| 资源 | 槽位 | 容量 | 说明 |
|------|------|:----:|------|
| 普通研发 | 1-6 | 6人 | 非绩效任务使用 |
| 绩效独立研发 | 101, 102 | 2人(动态共享) | 绩效独占；空闲时开放给普通任务 |

- 绩效（dev_slots="独立团队"）：仅使用 101+102，不占用 1-6
- 普通任务：引擎动态分配槽位，可借用空闲的 K1/K2
- K1/K2 共享：绩效无研发任务时自动纳入普通池

### 3.2 测试资源池（不变）

| 资源 | 槽位 | 容量 | 说明 |
|------|:----:|:----:|------|
| 测试 | 1-4 | 4人 | 所有任务共享 |

### 3.3 PM时间线（v4重构）

| 资源 | 说明 |
|------|------|
| 需求阶段 | 澄清+评审+需求设计，独占PM |
| 验收 | 产品验收，独占PM |
| 研发/测试 | **不占用PM时间**，可与PM任务并行 |

同一PM不可同时处理两件PM任务（需求阶段或验收）。研发/测试走研发/测试资源池，与PM任务不冲突。

## 四、输入规则（v4新增）

### 4.1 首Task（Task A）

PM只给定：
- 完整需求周期（clarify_start/end, tech_review_start/end, design_start/end）
- 技术评审日期（tech_review_A）

需求阶段开始时间：
```
如果 tech_review_A 倒推 req_wd_A 后 ≥ today:
  req_start_A = tech_review_A - req_wd_A（倒推）
否则:
  req_start_A = today → tech_review_A 顺延
```

### 4.2 后续Task（Task B/C/...）

PM不给定任何日期，全部由引擎生成：
- 需求阶段开始 = PM Scheduler 决定
- 技术评审 = 需求阶段结束日（联动后移）
- 研发/测试/验收 = 按引擎公式计算

## 五、排期算法

### 5.1 总流程：时间驱动逐日模拟

```
today = 指定日期
sim_day = today
while 有待排期任务:
    if sim_day 不是工作日: sim_day += 1; continue
    
    1. 检入：tech_review + 1wd <= sim_day → 进入研发就绪队列
    2. 研发阶段：为就绪任务分配研发资源（含K1/K2共享）
    3. dev→test：研发已结束 → 进入测试就绪队列
    4. 测试阶段：为就绪任务分配测试资源
    5. test→accept：测试已结束 → 验收就绪
    6. PM Scheduler（v4新增）：
       a. 检入就绪验收
       b. 如果PM空闲 → 收集待办→排序→取第一个执行
       c. 如果PM忙碌 → 不打断，等完成
    7. 全部完成 → 结束; 否则 sim_day += 1
```

### 5.2 研发阶段调度（不变）

```
就绪条件: tech_review + 1wd <= sim_day
处理顺序: 绩效 > 人事 > 考勤 > 薪酬 > 流程平台 > HRONE基础建设
```

### 5.3 测试阶段调度（不变）

```
就绪条件: 研发结束 + 1wd <= sim_day
软间隔约束(max-gap=5wd): 超时提升紧急度，不阻塞
```

### 5.4 PM Scheduler（v4核心新增）

```
在每个模拟日，对于每个PM：

① 检入：标记新就绪的验收(test_end + 1wd ≤ sim_day)

② 如果PM空闲:
   a. 收集所有待办（已就绪验收 + 未开始的需求阶段）
   b. 按"开始时间"排序（最早的先做）
   c. 开始时间相同 → 需求阶段优先
   d. 取第一个 → PM开始执行

③ 如果PM忙碌:
   a. 不打断当前任务
   b. 等当前任务完成后 → 回到 ②
```

### 5.5 验收就绪时的两种场景

```
                  ┌─ 1.1 PM空闲
验收A就绪 ──┤          验收A立刻开始
                  │
                  └─ 1.3 PM在做Task B的需求阶段
                        不打断B，等B需求结束(B技术评审)+1wd
                        → 验收A开始
```

### 5.6 传播效应

```
验收A占用PM时间 → Task B需求阶段后移
→ Task B技术评审后移（联动）
→ Task B研发开始后移（技术评审+1wd，公式不变）
→ 后续全部连锁后移
```

### 5.7 节假日（不变）

```
周六/周日 → 非工作日
法定假日 → 非工作日
调休补班 → 工作日（即使周末）
```

## 六、跨模块并行规则（不变）

| 场景 | 行为 |
|------|------|
| 绩效和考勤抢槽位1,2 | 绩效优先 |
| 绩效要1,2；考勤要3,4 | 同时排（不同槽位） |
| 人事要5,6且空闲 | 立即排，不等高优先级模块 |
| 同PM两个验收 | 串行，按开始时间排序 |

**核心原则**：优先级仅仲裁同槽竞争，空闲资源不囤积。

---

## 七、验证规则（v4更新）

```
1. 同一研发槽位无日期重叠
2. 同一测试槽位无日期重叠
3. 同一PM无验收重叠
4. PM验收不与需求阶段重叠
5. 研发开始 >= 技术评审 + 1wd
6. 测试开始 >= 研发结束 + 1wd
7. 验收开始 >= 测试结束 + 1wd
8. 需求阶段顺序：澄清≤澄清结束≤评审开始≤评审结束≤需求开始≤需求结束（v4新增）
9. 技术评审 = 需求阶段结束日（v4新增）
```

## 八、输出

### 报告
- 统计卡片（任务数、最终交付、资源冲突=0）
- 按模块展示每任务完整时间线（含需求阶段和技术评审）
- 研发/测试人员负载明细
- PM验收分布
- **PM需求阶段负载明细（v4新增）**

### 写回飞书（v4更新）
- 新增写回：澄清/评审/需求开始/结束(x6) + 技术评审
- 原有写回：研发/测试/验收开始/结束(x6) + 投入研发/测试 + 标准研发人天
- 空值处理：日期用 `1900-01-01`，槽位用 `未分配`
- **v4移除**：冲突后剩余设计人天（不再需要）
